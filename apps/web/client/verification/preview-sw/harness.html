preview-sw verification suite

Wave H §1.3 Standalone visual + DOM-assertion suite for the ExpoBrowser preview service worker. Each scenario exercises a different code path in preview-sw.js. Pass/fail is computed from DOM inspection; screenshots are saved as artifacts for human review.

Initializing scenarios…

1. Success pathpending

Push a valid bundle via postMessage, navigate iframe to /preview/sx-success/f1/, expect the IIFE to render into #root.

2. Error overlaypending

Push a bundle that throws at runtime, expect the IIFE wrapper to write the stack trace to document.body.

3. Placeholder before bundle arrivespending

Navigate iframe to /preview/sx-placeholder/f1/ WITHOUT pushing a bundle. The SW should serve the ‘waiting for first bundle…’ placeholder JS, the iframe shows the SW HTML shell’s loading message.

4. Multi-branch isolationpending

Push two different bundles for two different branchIds, two iframes side-by-side at /preview/sx-multi-A/f1/ and /preview/sx-multi-B/f1/. Each should render its own bundle independently.

5. Bundle update (live re-bundle)pending

Push bundle v1, load iframe, then push bundle v2 with different content, force iframe reload, expect v2 content. Verifies the SW cache is updated by subsequent postMessage calls.

6. HTML shell carries branch and frame metadatapending

Inspect the HTML shell’s <body> tag for data-branch-id and data-frame-id attributes set to the path segments. Confirms the shell reads the branchId/frameId from the URL.
