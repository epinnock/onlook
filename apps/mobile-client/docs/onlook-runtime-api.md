# OnlookRuntime JSI API

## Overview

OnlookRuntime is a C++ JSI host object registered on `globalThis.OnlookRuntime`
at app startup by the platform-specific installer TurboModule
(`OnlookRuntimeInstaller.mm` on iOS, `onlook_runtime_installer.cpp` on Android).
The object is read-only from JS -- assignments throw a `jsi::JSError`.

One instance exists per Hermes runtime. All methods execute on the JS thread;
native callers on other threads must hop via `RCTBridge.runOnJSThread` (iOS) or
the CallInvoker (Android).

Source: `apps/mobile-client/cpp/OnlookRuntime.h`,
`apps/mobile-client/cpp/OnlookRuntime.cpp`.

## Properties and methods

### `runApplication(bundleSource: string, props?: object): void`

Evaluates `bundleSource` as JavaScript in the host Hermes runtime (single-context
model per ADR `plans/adr/MC1.4-MC2.10-runtime-context.md`), then calls
`globalThis.onlookMount(props)`. If `props` is omitted an empty object is passed.

Throws `jsi::JSError` when:

- `bundleSource` is not a string (arg-validation).
- The bundle fails to evaluate (rewrapped with a `bundle eval failed:` prefix).
- `globalThis.onlookMount` is not defined or not callable after eval.

All throws are caught by `captureAndReport` and funneled through
`dispatchEvent('onlook:error', ...)` so callers observe errors via the event bus
rather than an uncaught exception.

Source: `apps/mobile-client/cpp/OnlookRuntime_runApplication.cpp` (MC2.7).

### `reloadBundle(bundleSource: string): void`

Atomic teardown-then-remount cycle used by the relay's hot-reload path:

1. Calls `globalThis.onlookUnmount()` if present. Teardown failures are caught
   and swallowed -- they do not surface as `onlook:error` events and do not
   block the remount.
2. Forwards to `runApplicationImpl(bundleSource)` to re-evaluate and re-mount.

Props from the previous `runApplication` call are not currently replayed (tracked
as an MC2.8 follow-up using `mountedBundleSha_` and a future `mountedProps_`
field). A byte-equal short-circuit (idempotent under WS double-fire) is planned
via SHA-256 comparison of `bundleSource`.

Source: `apps/mobile-client/cpp/OnlookRuntime_reloadBundle.cpp` (MC2.8).

### `dispatchEvent(name: string, payload?: any): void`

Posts an event to JS-side listeners registered on
`globalThis.__onlookEventBus.dispatch(name, payload)`. The event bus is installed
by the runtime shell (`shell.js`).

Behavior when the bus is missing or not yet installed:

- **Bus absent** (`__onlookEventBus` not an object): benign no-op. A breadcrumb
  is logged via `nativeLoggingHook` (level 1) if available.
- **Bus present but `.dispatch` not callable**: throws `jsi::JSError` -- this is
  a protocol violation.

Payload is passed through as-is at the JSI boundary (not JSON-serialized in C++;
serialization, if any, is handled by the JS-side bus).

Known events dispatched by the runtime:

| Event name        | Producer            |
|-------------------|---------------------|
| `onlook:error`    | `reportRuntimeError` (MC2.14) |
| `onlook:tap`      | Tap forwarder (MC4.2) |
| `onlook:select`   | Inspector (MC4.x) |

Source: `apps/mobile-client/cpp/OnlookRuntime_dispatchEvent.cpp` (MC2.9).

### `version: string` (read-only getter)

Returns the compile-time `ONLOOK_RUNTIME_VERSION_STRING` macro baked into the
binary. The macro is generated by
`apps/mobile-client/scripts/generate-version-header.ts` from the
`@onlook/mobile-client-protocol` TypeScript constant and written to
`OnlookRuntime_version.generated.h`.

The relay's manifest builder compares this value against
`extra.expoClient.onlookRuntimeVersion` in the manifest to refuse mismatched
bundles (MC6.2).

Currently frozen at `v0.1.0`.

Source: `apps/mobile-client/cpp/OnlookRuntime_version.cpp` (MC2.12).

## Error surface

Defined in `apps/mobile-client/cpp/OnlookRuntime_errorSurface.cpp` (MC2.14).

### `reportRuntimeError(rt, kind, message, stack)`

Posts `{ kind, message, stack }` to
`globalThis.OnlookRuntime.dispatchEvent('onlook:error', payload)`. Safe to call
before OnlookRuntime has been installed -- silently no-ops. The function itself
never throws (outer `catch(...)` swallows).

### `captureAndReport(rt, fn)`

Runs `fn()` in a try/catch and converts thrown exceptions into
`reportRuntimeError` calls:

| Exception type     | `kind`      |
|--------------------|-------------|
| `jsi::JSError`     | `"js"`      |
| `std::exception`   | `"native"`  |
| anything else      | `"unknown"` |

Used internally by `runApplication`, `reloadBundle`, and `dispatchEvent` to
ensure all unhandled errors funnel through the single `onlook:error` event.

### `prewarmInspector(rt)`

Pre-warms Fabric's `findNodeAtPoint` with off-screen coordinates `(-1, -1)` to
absorb the ~150 ms cold-start latency during mount. Called by the installer right
after `globalThis.OnlookRuntime` is wired up. Best-effort: silently no-ops if
`nativeFabricUIManager` or `findNodeAtPoint` are missing.

Source: `apps/mobile-client/cpp/InspectorPrewarm.cpp` (MC2.15).

## OnlookInspector (separate host object)

Installed on `globalThis.OnlookInspector` by a separate platform installer
(MC4.1). Read-only from JS, same lifetime model as OnlookRuntime.

Source: `apps/mobile-client/cpp/OnlookInspector.h`.

### `captureTap(x: number, y: number): { reactTag: number, ... }`

Returns the topmost Fabric component under screen-space point `(x, y)` via
`nativeFabricUIManager.findNodeAtPoint`. Throws on invalid coordinates or no hit.

Source: MC4.2.

### `walkTree(reactTag: number): { reactTag, componentName, children[] }`

Recursively walks the shadow tree rooted at `reactTag` and returns a plain-JS
snapshot for the editor's element tree panel.

Source: MC4.3.

### `captureScreenshot(): string`

Returns a base64-encoded PNG of the current key window. Uses
`UIGraphicsImageRenderer` on iOS, `PixelCopy.request` on Android.

Source: MC4.4.

### `highlightNode(reactTag: number, color: string): void`

Draws a 2 pt overlay border on the native view backing `reactTag` in `color`
(hex string, e.g. `"#FF0000"`), fading out over 600 ms. Pass `reactTag === 0`
to clear all highlights.

Source: MC4.5.
